## PART 1: The App — Local-First Architecture

### 1.1 Core Principle

All user data lives on the device by default. No account required to get full value. An optional Convex backend enables cross-device sync, notifications, and advanced features — but the app must be 100% functional with zero network connectivity and zero backend dependency.

### 1.3 Convex Backend (Optional)

When the user opts in to sync, a Convex schema is deployed with the following tables:

```typescript
// convex/schema.ts
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({
  users: defineTable({
    convexUserId: v.id("_users"),    // Convex auth
    email: v.optional(v.string()),
    name: v.string(),
    preferences: v.object({
      theme: v.optional(v.union(v.literal("light"), v.literal("dark"), v.literal("system"))),
      hideMetricsByDefault: v.optional(v.boolean()),
      language: v.optional(v.union(v.literal("no"), v.literal("en"))),
      remindersEnabled: v.optional(v.boolean()),
      reminderTime: v.optional(v.string()),
      weeklySummaryEnabled: v.optional(v.boolean()),
      preferredSessionTypes: v.optional(v.array(v.string())),
      physicalConstraints: v.optional(v.array(v.string())),
    }),
    createdAt: v.number(),
    lastSyncedAt: v.number(),
  }).index("by_convex_user_id", ["convexUserId"]),

  syncEvents: defineTable({
    userId: v.id("users"),
    deviceId: v.string(),
    entityType: v.union(
      v.literal("dayLog"), v.literal("movement"),
      v.literal("win"), v.literal("care"),
      v.literal("celebration"), v.literal("profile"),
      v.literal("preferences"), v.literal("programEnrollment"),
      v.literal("reflection")
    ),
    entityId: v.string(),
    operation: v.union(v.literal("upsert"), v.literal("delete")),
    payload: v.string(),         // JSON stringified
    timestamp: v.number(),
    clientVersion: v.number(),
  }).index("by_user", ["userId"])
    .index("by_user_entity", ["userId", "entityType"])
    .index("by_user_timestamp", ["userId", "timestamp"]),

  programs: defineTable({
    userId: v.id("users"),
    programId: v.string(),
    startedAt: v.number(),
    currentDay: v.number(),
    completedDays: v.array(v.number()),
    droppedAt: v.optional(v.number()),
  }).index("by_user", ["userId"]),

  reflections: defineTable({
    userId: v.id("users"),
    weekStart: v.string(),
    answeredAt: v.number(),
    responses: v.string(),        // JSON
  }).index("by_user", ["userId"])
    .index("by_user_week", ["userId", "weekStart"]),

  devices: defineTable({
    userId: v.id("users"),
    deviceId: v.string(),
    lastSyncedAt: v.number(),
    platform: v.optional(v.string()),
    appVersion: v.string(),
  }).index("by_user", ["userId"])
    .index("by_user_device", ["userId", "deviceId"]),
});
```

**Sync protocol:**
1. Client generates a `deviceId` (UUID stored in IndexedDB).
2. On each state change, a `syncEvent` row is written locally with a monotonic client timestamp.
3. On connect, client sends its latest known server timestamp.
4. Server returns all events after that timestamp for this user.
5. Client applies events in timestamp order, merging with local state (last-write-wins per entityId).
6. Server stores Convex-generated events (e.g. server-initiated weekly summary generation).

**Recommendation engine logic (local, no backend required):**

```typescript
function getRecommendation(state: AppState, today: DayLog): RecommendedAction {
  const energy = today.energy ?? 3;
  const streak = state.streakData.currentContinuity;
  const recentSessions = state.movement.sessions.slice(0, 7);
  const careThisWeek = state.care.slice(0, 7).length;
  const lastGrace = getLastGraceDay(state.dayLog);
  const daysSinceGrace = lastGrace ? daysBetween(lastGrace, today.date) : 999;

  // Rule 1: If energy is 1-2, always suggest rest or micro-wins
  if (energy <= 2) {
    return {
      type: 'micro',
      label: getMicroOption(energy, state.preferences),
      reason: 'Din energi er lav. Et minutt med omsorg teller.',
      icon: 'omsorg',
      cta: 'Logg en liten seier',
    };
  }

  // Rule 2: If streak is fragile (< 3 days), suggest showing up
  if (streak < 3) {
    return {
      type: 'momentum',
      label: 'Kom i gang igjen med én liten handling',
      reason: 'Hvert skritt gjenoppretter rytmen din.',
      icon: 'bevegelse',
      cta: 'Start en rolig økt',
    };
  }

  // Rule 3: If no movement logged today, suggest based on energy band
  if (!hasLoggedMovementToday(state)) {
    const sessionType = SESSION_TYPES_BY_ENERGY[energy] ?? 'Rolig gåtur';
    return {
      type: 'movement',
      label: sessionType,
      reason: getReasonForEnergy(energy, recentSessions),
      icon: 'bevegelse',
      cta: `Start ${sessionType}`,
    };
  }

  // Rule 4: If care score is low this week
  if (careThisWeek < 3) {
    return {
      type: 'care',
      label: 'En omsorgshandling i dag?',
      reason: 'Omsorg for deg selv er også fremgang.',
      icon: 'omsorg',
      cta: 'Logg omsorg',
    };
  }

  // Rule 5: Grace day promotion if streak is strong and user hasn't used grace recently
  if (streak >= 7 && daysSinceGrace > 14) {
    return {
      type: 'grace',
      label: 'Du har fortjent en hviledag når som helst',
      reason: 'Ingen grunn til å vente på perfekte forhold.',
      icon: 'hvile',
      cta: 'Ta en nådedag',
    };
  }

  // Default: gentle encouragement to continue
  return {
    type: 'celebrate',
    label: 'Du er på vei',
    reason: `Du har ${streak} dager med kontinuitet. Det er ekte fremgang.`,
    icon: 'seier',
    cta: 'Seire i dag',
  };
}
```

**Session flow:**
1. Select type
2. Optional: set an intention ("intensjon")
3. Rate mood before (1-5)
4. Optional: hide metrics toggle
5. Start timer (live display of elapsed time in mm:ss)
6. At any time: pause / resume / stop
7. On stop: rate effort (1-5), mood after (1-5), add optional notes
8. Auto-log a win "Jeg møtte opp for meg selv"
9. Trigger celebration modal

**Quiet mode:**
- When `hideMetrics` is true: timer shows no numbers, only a breathing animation and "Du er i bevegelse" text.
- Effort/mood ratings can still be done.

**Session history:**
- Chronological list with date grouping
- Filter by type, energy band, or date range
- Each entry shows: type, duration, mood delta (before → after), effort, optional notes

## PART 8: Design System

### 8.1 Visual Language

**Primary palette (light mode):**
- Background: `#fbfbe2` (warm oat)
- Surface: `#efefd7`
- Primary: `#526442` (dusty sage green)
- Secondary: `#9f402d` (soft terracotta)
- Tertiary: `#74593f` (warm brown)
- Text: `#1b1d0e`
- Text soft: `#44483f`

**Dark mode equivalents:**
- Background: `#1d2117`
- Surface: `#273022`
- Primary: `#b9cda4`
- Secondary: `#ffb4a5`
- Text: `#f2f2d9`

**Typography:**
- Headlines: Plus Jakarta Sans (Google Fonts), weight 700–800
- Body: Be Vietnam Pro (Google Fonts), weight 400–600
- Monospace (data): JetBrains Mono (for timers, metrics)

**Spatial system:**
- Base unit: 4px
- Spacing scale: 4, 8, 12, 16, 24, 32, 48, 64px
- Border radius: 18px (sm), 28px (md), 42px (lg), 999px (pill)

**Motion principles:**
- Duration: 150ms (micro), 250ms (standard), 400ms (entrance)
- Easing: `cubic-bezier(0.4, 0, 0.2, 1)` for most, spring-like for celebrations
- No motion that feels urgent or alarming
- Breathing animations for quiet mode (slow, 4s cycles)

## PART 11: Decisions Made By This Mission (No User Input Required)

These are the definitive choices the mission subagent will implement without further input:

1. **Framework:** Next.js 14 App Router, TypeScript, static export for marketing, Node.js for app
2. **State management:** Zustand for UI state, IndexedDB (Dexie.js) for local persistence
3. **Backend:** Convex for optional cloud sync (auth + data + scheduled functions)
4. **Styling:** CSS Modules with CSS custom properties (no Tailwind, keep existing design system)
5. **PWA:** Workbox for service worker, next-pwa for manifest injection
6. **Analytics:** Plausible Analytics (self-hosted or .com, privacy-first)
7. **Email:** ConvertKit for newsletter (free tier, or Brevo for EU-compliant free tier)
8. **Blog:** MDX files in the repo, rendered via Next.js App Router
9. **Programs:** JSON files in repo, no CMS for v1
10. **Icons:** Phosphor Icons (MIT, consistent)
11. **Fonts:** Plus Jakarta Sans + Be Vietnam Pro (already in use, keep)
12. **Content language:** Norwegian primary, English secondary (all content in both)
13. **Pricing:** Always free for all core features, forever
14. **Monetization signal:** Display a single "Støtt arbeidet" (support this work) link to Ko-fi or GitHub Sponsors in settings and footer — no paywall
15. **App distribution:** Web-first (PWA installable). App Store / Google Play optional post-launch.
16. **Social platforms:** Instagram (primary), TikTok (secondary), LinkedIn (tertiary)
17. **Community:** Discord server (free, easy moderation)
18. **Hosting:** Vercel (marketing + app shell), Convex (backend)
19. **Domain:** user-decided, but recommended pattern: `[name].no` or `[name].app`
20. **Launch order:** Soft launch (website + PWA) before any app store submission
21. **Blog cadence:** 1 post/week, maintained indefinitely post-launch
22. **Social cadence:** 5 Instagram + 3 TikTok + 1 LinkedIn per week
23. **Analytics review cadence:** Weekly during first 3 months, then monthly
24. **Retention target (12 weeks post-launch):** D30 retention > 15% for a free wellness app is good; aim for 20%+
25. **Scope boundary:** No AI-generated content in the app itself (all content is human-curated). AI is used only in the recommendation engine logic and build tooling.
